# 侧边栏导航结构重构 - 资源下载页面合并计划

## 1. 现状分析

### 1.1 侧边栏导航结构
当前侧边栏包含以下项目：
- 启动 (Launch)
- 下载版本 (Download Version)
- mods
- 版本列表 (Version List)
- 设置 (Settings)

### 1.2 现有页面架构
- **下载Page.xaml**：用于下载Minecraft版本
- **ModPage.xaml**：用于下载mods
- **版本列表Page.xaml**：用于管理版本
- **版本管理Page.xaml**：用于管理特定版本的mods、光影、资源包等

### 1.3 技术栈
- WinUI 3
- MVVM模式
- 依赖注入
- NavigationView + Frame导航

## 2. 重构目标

### 2.1 核心需求
- 合并版本和mod页面为统一的"资源下载"页面
- 使用TabView控件实现不同资源类型的切换
- 支持版本、mod、光影、资源包下载
- 保持现有功能完整性
- 确保导航结构清晰可持续

### 2.2 预期效果
- 侧边栏只保留核心导航项：启动、资源下载、版本列表、设置
- 资源下载页面内部通过TabView切换不同资源类型
- 为未来扩展预留空间

## 3. 实现方案

### 3.1 新页面创建
- **ResourceDownloadPage.xaml**：统一的资源下载页面
- **ResourceDownloadViewModel.cs**：对应的ViewModel

### 3.2 页面结构设计
```
ResourceDownloadPage
├── TabView (顶部对齐，居中)
│   ├── TabItem：版本下载
│   │   └── 原下载Page内容
│   ├── TabItem：Mod下载
│   │   └── 原ModPage内容
│   ├── TabItem：光影下载
│   │   └── 占位符内容
│   └── TabItem：资源包下载
│       └── 占位符内容
└── 其他公共组件
```

### 3.3 迁移步骤

#### 3.3.1 步骤1：创建新页面和ViewModel
- 创建ResourceDownloadPage.xaml和ResourceDownloadViewModel.cs
- 在App.xaml.cs中注册新页面和ViewModel

#### 3.3.2 步骤2：设计TabView布局
- 在ResourceDownloadPage.xaml中添加TabView控件
- 配置TabView的样式和布局
- 添加四个TabItem：版本、mod、光影、资源包

#### 3.3.3 步骤3：迁移版本下载功能
- 将下载Page.xaml的XAML内容复制到版本TabItem中
- 确保数据绑定和命令正确指向ResourceDownloadViewModel
- 调整布局以适应TabView容器

#### 3.3.4 步骤4：迁移Mod下载功能
- 将ModPage.xaml的XAML内容复制到mod TabItem中
- 确保数据绑定和命令正确指向ResourceDownloadViewModel
- 调整布局以适应TabView容器

#### 3.3.5 步骤5：创建光影和资源包占位符
- 为光影和资源包TabItem添加基础布局
- 添加"功能开发中"提示

#### 3.3.6 步骤6：更新导航结构
- 在ShellPage.xaml中，将原有的"下载"和"mod"导航项替换为"资源下载"导航项
- 更新NavigationHelper.NavigateTo属性指向ResourceDownloadViewModel

#### 3.3.7 步骤7：测试和验证
- 确保所有现有功能正常工作
- 测试导航和页面切换
- 验证数据流动和API调用

### 3.4 关键技术点

#### 3.4.1 TabView控件使用
- 使用TabView.SelectedIndex或TabView.SelectedItem跟踪当前选中的标签
- 处理TabView.SelectionChanged事件
- 为每个TabItem设置Header和Content

#### 3.4.2 数据绑定
- 确保原有的数据服务和API调用在新页面中正常工作
- 调整命令绑定，确保命令正确执行

#### 3.4.3 导航服务
- 确保新页面正确注册到导航服务
- 处理页面间的导航和参数传递

## 4. 风险评估

### 4.1 潜在风险
- 功能迁移过程中可能出现数据绑定错误
- 导航结构变更可能导致页面无法访问
- 现有API调用可能需要调整

### 4.2 风险缓解措施
- 详细记录现有代码结构和数据流程
- 分步迁移，每步进行测试
- 保留原有页面作为备份，以便回滚
- 确保所有依赖注入注册正确
